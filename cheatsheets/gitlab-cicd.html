Structure .gitlab-ci.yml

Fichier YAML placé à la racine du repo. GitLab le lit à chaque push.

YAML
# Ordre d'exécution global des stages
stages:
  - build
  - test
  - deploy

# Variables globales (disponibles dans tous les jobs)
variables:
  APP_NAME: "my-app"
  DOCKER_IMAGE: "$CI_REGISTRY_IMAGE:$CI_COMMIT_SHORT_SHA"

# Image Docker par défaut pour tous les jobs
default:
  image: node:20-alpine
  before_script:
    - npm ci --cache .npm

# Un job minimal
build-app:
  stage: build
  script:
    - npm run build
  artifacts:
    paths:
      - dist/
Variables prédéfinies clés
VariableValeur
CI_COMMIT_SHAHash complet du commit
CI_COMMIT_SHORT_SHA8 premiers caractères du hash
CI_COMMIT_BRANCHNom de la branche
CI_COMMIT_TAGTag git (si pipeline déclenché par tag)
CI_PIPELINE_IDID unique du pipeline
CI_JOB_NAMENom du job en cours
CI_REGISTRYURL du registry Docker GitLab
CI_REGISTRY_IMAGEImage de base du projet
CI_REGISTRY_USER / PASSWORDCredentials registry CI
CI_PROJECT_DIRRépertoire de travail du runner
CI_ENVIRONMENT_NAMENom de l'environnement déployé

Stages & Jobs

YAML
stages:
  - build
  - test
  - release
  - deploy

# Job complet avec toutes les options courantes
unit-tests:
  stage: test
  image: node:20-alpine       # surcharge l'image default
  tags:                        # cibler un runner spécifique
    - docker
  allow_failure: false         # true = job rouge mais pipeline continue
  timeout: 10 minutes
  retry:
    max: 2
    when:
      - runner_system_failure
      - stuck_or_timeout_failure
  before_script:
    - npm ci
  script:
    - npm test -- --coverage
  after_script:
    - echo "Tests terminés"
  coverage: '/Lines\s*:\s*(\d+\.?\d*)%/'  # regex extraction coverage

# Job déclenché manuellement
deploy-prod:
  stage: deploy
  script:
    - ./deploy.sh
  when: manual           # on_success | on_failure | always | manual | delayed
  start_in: "1 hour"    # uniquement pour when: delayed
Valeurs de `when`
ValeurComportement
on_successDéfaut — s'exécute si tous les jobs précédents réussissent
on_failureS'exécute uniquement si un job précédent échoue
alwaysS'exécute quoi qu'il arrive (utile pour nettoyage)
manualNécessite un clic humain dans l'UI GitLab
delayedAttend un délai configuré via start_in
neverNe s'exécute jamais (utile avec rules pour désactiver)

Variables

YAML
# Variables globales
variables:
  ENV: "production"
  PORT: "3000"                 # toujours une string en YAML
  URL: "https://$CI_PROJECT_NAME.example.com"  # interpolation

# Variable locale à un job (surcharge la globale)
my-job:
  variables:
    ENV: "staging"
    GIT_STRATEGY: none      # fetch | clone | none
    GIT_DEPTH: "10"          # shallow clone pour accélérer
  script:
    - echo $ENV              # → staging
    - echo ${ENV:-default}   # valeur par défaut bash

# Secrets — à définir dans GitLab UI (Settings > CI/CD > Variables)
# Type "Masked" = caché dans les logs
# Type "Protected" = uniquement sur branches protégées
deploy:
  script:
    - echo $DB_PASSWORD      # variable masquée définie dans l'UI
    - curl -H "Token: $API_KEY" https://api.example.com
Priorité des variables (haute → basse)
SourceScope
Trigger variablesVariables passées au déclenchement via API
Variables UI projet (protected)Branches/tags protégés uniquement
Variables UI projetTous les pipelines du projet
Variables UI groupeTous les projets du groupe
Variables .gitlab-ci.yml (job)Job courant
Variables .gitlab-ci.yml (global)Tous les jobs
Variables prédéfinies CICI_*, GITLAB_*, etc.

Rules & conditions

rules remplace only/except (déprécié). Plus flexible et lisible.

YAML
deploy-staging:
  script: ./deploy.sh staging
  rules:
    # Si branche develop → exécute automatiquement
    - if: '$CI_COMMIT_BRANCH == "develop"'
      when: on_success
    # Si MR vers main → exécute manuellement
    - if: '$CI_MERGE_REQUEST_TARGET_BRANCH_NAME == "main"'
      when: manual
    # Si fichier modifié
    - changes:
        - src/**/*
        - package.json
    # Sinon, ne pas exécuter
    - when: never

release:
  script: ./release.sh
  rules:
    # Uniquement sur les tags semver
    - if: '$CI_COMMIT_TAG =~ /^v\d+\.\d+\.\d+$/'

# Variables injectées selon la règle
build:
  script: npm run build
  rules:
    - if: '$CI_COMMIT_BRANCH == "main"'
      variables:
        BUILD_ENV: production
    - if: '$CI_COMMIT_BRANCH'
      variables:
        BUILD_ENV: staging
Conditions `if` courantes
ConditionCas d'usage
$CI_COMMIT_BRANCH == "main"Uniquement sur main
$CI_COMMIT_BRANCH =~ /^feature/Toutes les branches feature/*
$CI_COMMIT_TAGSi un tag est poussé
$CI_PIPELINE_SOURCE == "merge_request_event"Pipeline MR uniquement
$CI_PIPELINE_SOURCE == "schedule"Pipeline planifié (cron)
$CI_PIPELINE_SOURCE == "web"Déclenché depuis l'UI
$MY_VAR == nullVariable non définie

Needs & DAG

needs permet un pipeline en graphe orienté acyclique (DAG) — les jobs ne se bloquent plus par stage.

YAML
stages:
  - build
  - test
  - deploy

build-frontend:
  stage: build
  script: npm run build:frontend

build-backend:
  stage: build
  script: npm run build:backend

# Ne attend que build-frontend, pas build-backend
test-frontend:
  stage: test
  needs:
    - job: build-frontend
      artifacts: true       # télécharge les artifacts du job parent
  script: npm run test:frontend

# Ne attend que build-backend
test-backend:
  stage: test
  needs: [build-backend]
  script: npm run test:backend

# Attend les deux jobs de test
deploy-staging:
  stage: deploy
  needs: [test-frontend, test-backend]
  script: ./deploy.sh

# needs: [] = démarre immédiatement (ignore les stages)
notify-start:
  stage: test
  needs: []
  script: curl -X POST $SLACK_WEBHOOK -d '{"text":"Pipeline démarré"}'

Artifacts & Cache

Artifacts — partager des fichiers entre jobs
YAML
build:
  script: npm run build
  artifacts:
    paths:
      - dist/
      - coverage/
    exclude:
      - dist/**/*.map              # exclure les sourcemaps
    reports:
      junit: test-results.xml     # rapport visible dans l'UI MR
      coverage_report:
        coverage_format: cobertura
        path: coverage/cobertura.xml
    expire_in: 1 week            # 30 mins | 1 day | 1 week | never
    when: always                  # conserver même si le job échoue
Cache — accélérer les jobs (node_modules, pip…)
YAML
install:
  script: npm ci
  cache:
    key:
      files:                       # clé basée sur le hash du fichier
        - package-lock.json
    paths:
      - node_modules/
      - .npm/
    policy: pull-push             # pull | push | pull-push (défaut)

# Cache partagé entre branches (key fixe)
test:
  cache:
    key: "$CI_COMMIT_REF_SLUG-node"
    paths: [node_modules/]
    policy: pull                  # ne pas re-pousser le cache

# Différence artifacts vs cache :
# artifacts = entre jobs du même pipeline (garanti)
# cache = entre pipelines (best-effort, peut être absent)

Runners

YAML
# Cibler un runner par tag
build-docker:
  tags:
    - docker
    - linux

# Cibler un runner par tag (opérateur OU)
deploy-k8s:
  tags:
    - kubernetes

# Runner shell (pas de conteneur)
legacy-build:
  tags:
    - shell
  script:
    - make build
Types d'exécuteur (executor)
ExecutorDescription
shellExécute directement sur le serveur hôte
dockerChaque job dans un conteneur Docker isolé
docker+machineAuto-scale via Docker Machine (cloud)
kubernetesPods Kubernetes éphémères par job
virtualbox / parallelsVMs isolées (rare)
BASH — Enregistrer un runner
# Installation du runner
curl -L https://packages.gitlab.com/install/repositories/runner/gitlab-runner/script.deb.sh | sudo bash
sudo apt install gitlab-runner

# Enregistrement interactif
sudo gitlab-runner register

# Enregistrement non-interactif
sudo gitlab-runner register \
  --non-interactive \
  --url "https://gitlab.com/" \
  --registration-token "$RUNNER_TOKEN" \
  --executor "docker" \
  --docker-image "alpine:latest" \
  --description "mon-runner" \
  --tag-list "docker,linux"

Services (Docker, BDD)

Les services lancent des conteneurs accessibles via leur alias hostname.

YAML
tests-integration:
  image: node:20-alpine
  services:
    - name: postgres:16-alpine
      alias: db                  # accès via hostname "db"
    - name: redis:7-alpine
      alias: redis
  variables:
    POSTGRES_DB: testdb
    POSTGRES_USER: ci
    POSTGRES_PASSWORD: ci
    DATABASE_URL: "postgresql://ci:ci@db:5432/testdb"
  script:
    - npm ci
    - npm run db:migrate
    - npm run test:integration

# Build d'image Docker (Docker-in-Docker)
build-image:
  image: docker:24
  services:
    - docker:24-dind              # Docker-in-Docker
  variables:
    DOCKER_TLS_CERTDIR: "/certs"
  script:
    - docker login -u $CI_REGISTRY_USER -p $CI_REGISTRY_PASSWORD $CI_REGISTRY
    - docker build -t $CI_REGISTRY_IMAGE:$CI_COMMIT_SHORT_SHA .
    - docker push $CI_REGISTRY_IMAGE:$CI_COMMIT_SHORT_SHA

Environments & Deploy

YAML
deploy-staging:
  stage: deploy
  script:
    - ./deploy.sh $CI_ENVIRONMENT_URL
  environment:
    name: staging
    url: https://staging.example.com
    on_stop: stop-staging         # job à appeler pour arrêter l'env

stop-staging:
  stage: deploy
  script:
    - ./teardown.sh staging
  environment:
    name: staging
    action: stop               # stop | start | prepare | verify | access
  when: manual

# Environnement dynamique par branche (Review Apps)
deploy-review:
  stage: deploy
  script:
    - ./deploy-review.sh $CI_COMMIT_REF_SLUG
  environment:
    name: review/$CI_COMMIT_REF_SLUG
    url: https://$CI_COMMIT_REF_SLUG.review.example.com
    on_stop: stop-review
    auto_stop_in: 1 week
  rules:
    - if: $CI_PIPELINE_SOURCE == "merge_request_event"

deploy-prod:
  stage: deploy
  script:
    - ./deploy.sh production
  environment:
    name: production
    url: https://example.com
    deployment_tier: production  # development | staging | production | …
  rules:
    - if: '$CI_COMMIT_TAG =~ /^v\d+/'
      when: manual

Extends & Anchors YAML

extends — héritage de jobs
YAML
# Job de base (préfixe . = caché, non exécuté)
.base-node:
  image: node:20-alpine
  before_script:
    - npm ci --cache .npm
  cache:
    key: "$CI_COMMIT_REF_SLUG"
    paths: [.npm/]

build:
  extends: .base-node          # hérite de .base-node
  stage: build
  script: npm run build

test:
  extends: .base-node
  stage: test
  script: npm test

# extends multiple (liste)
lint:
  extends:
    - .base-node
    - .rules-mr-only
Anchors YAML — réutilisation native YAML
YAML
# Définir un ancre
.before_npm: &before_npm
  - npm ci
  - echo "Dépendances installées"

# Réutiliser l'ancre
build:
  before_script: *before_npm
  script: npm run build

# Merge de maps avec <<
.common: &common
  image: node:20
  tags: [docker]

my-job:
  <<: *common              # merge les clés de .common
  script: echo "hello"

# Nota : extends est préférable aux anchors YAML
# car extends fusionne récursivement (anchors non)

Include & Templates

YAML
# Inclure des fichiers YAML depuis différentes sources
include:

  - local: ".gitlab/ci/build.yml"

  # Template officiel GitLab
  - template: Auto-DevOps.gitlab-ci.yml

  - project: my-group/ci-templates
    ref: main
    file: templates/docker.yml

# Surcharger un job inclus
SAST:
  variables:
    SAST_EXCLUDED_PATHS: "spec,test,tests,tmp"
  rules:
    - if: $CI_PIPELINE_SOURCE != "schedule"

Parallel matrix

YAML
# Lancer N copies identiques d'un job
rspec:
  script: bundle exec rspec
  parallel: 5                   # 5 jobs identiques en parallèle

# Matrix — combinatoire de variables
test-matrix:
  image: "$RUNTIME:$VERSION"
  script: npm test
  parallel:
    matrix:
      - RUNTIME: node
        VERSION: ["18", "20", "22"]
      - RUNTIME: bun
        VERSION: ["1.0", "1.1"]
  # → 5 jobs : node:18, node:20, node:22, bun:1.0, bun:1.1

# Matrix pour déploiement multi-régions
deploy:
  script:
    - ./deploy.sh --region $REGION --env $ENV
  parallel:
    matrix:
      - REGION: [eu-west-1, us-east-1]
        ENV: [staging, production]
  # → 4 jobs (2 régions × 2 envs)

Pipelines parent / enfant

YAML — Pipeline parent
# .gitlab-ci.yml (pipeline parent)
stages:
  - triggers

trigger-frontend:
  stage: triggers
  trigger:
    include: frontend/.gitlab-ci.yml  # pipeline enfant local
    strategy: depend                  # parent attend l'enfant
  rules:
    - changes: [frontend/**/*]

trigger-api:
  trigger:
    project: my-group/api-service   # pipeline dans un autre repo
    branch: main
  rules:
    - changes: [api/**/*]
YAML — Pipeline dynamique (généré)
# Job qui génère un fichier YAML dynamiquement
generate-pipeline:
  script:
    - python3 scripts/gen_pipeline.py > generated-pipeline.yml
  artifacts:
    paths: [generated-pipeline.yml]

run-generated:
  trigger:
    include:
      - artifact: generated-pipeline.yml
        job: generate-pipeline
    strategy: depend
  needs: [generate-pipeline]

Security scans (SAST, DAST…)

YAML
# Activer tous les scanners de sécurité GitLab
include:
  - template: Jobs/SAST.gitlab-ci.yml
  - template: Jobs/Secret-Detection.gitlab-ci.yml
  - template: Jobs/Dependency-Scanning.gitlab-ci.yml
  - template: Jobs/Container-Scanning.gitlab-ci.yml

# Configurer SAST
semgrep-sast:
  variables:
    SAST_EXCLUDED_PATHS: "spec,test,tests,tmp,node_modules"

# DAST — scan de l'appli déployée
include:
  - template: DAST.gitlab-ci.yml

dast:
  variables:
    DAST_WEBSITE: https://staging.example.com
    DAST_FULL_SCAN_ENABLED: "true"

# Scan d'image Docker
container_scanning:
  variables:
    CS_IMAGE: $CI_REGISTRY_IMAGE:$CI_COMMIT_SHORT_SHA
Rapport de vulnérabilités

Les résultats apparaissent automatiquement dans l'onglet Security de la MR et dans le tableau de bord Security > Vulnerability Report du projet.

Bonnes pratiques

YAML — Workflow recommandé
# Limiter les pipelines inutiles
workflow:
  rules:
    - if: $CI_PIPELINE_SOURCE == "merge_request_event"
    - if: $CI_COMMIT_BRANCH == $CI_DEFAULT_BRANCH
    - if: $CI_COMMIT_TAG
    - when: never                  # évite les pipelines en double MR+push

# Valider le YAML localement
# → GitLab UI : CI/CD > Pipelines > CI Lint
# → CLI : gitlab-runner exec docker <job-name>

# Réduire la taille des images
.slim-defaults:
  image: alpine:3.19           # préférer :alpine ou :slim
  variables:
    GIT_DEPTH: "1"             # shallow clone
    FF_USE_FASTZIP: "true"     # compression artifacts plus rapide
